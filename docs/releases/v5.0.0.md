# Kap v5

Tag: `v5.0.0`

Kap v5 modernizes and repairs the application while preserving the lightweight menu-bar workflow: click the tray, choose a region, record, and export.

## Downloads

- Apple Silicon - M1, M2, M3, M4, and newer: download `Kap-5.0.0-mac-arm64.dmg`
- Intel Mac: download `Kap-5.0.0-mac-x64.dmg`

Open **Apple menu -> About This Mac** if you are unsure which architecture your Mac uses. Macs that show **Chip** are Apple Silicon. Macs that show an Intel **Processor** use the x64 build.

Runtime verification for this release was performed on an Intel Mac. The Apple Silicon artifact is built and statically verified for arm64; feedback from M1, M2, M3, M4, and newer Macs is especially appreciated.

## Highlights

- Repaired the cropper overlay startup path for packaged and development builds.
- Restored renderer asset loading for static Next.js exports.
- Fixed cropper interaction readiness so transparent windows do not accept input before controls have painted.
- Reset stale crop drag, resize, and pick state across close and reopen cycles.
- Fixed start and stop recording shortcuts so they control recording rather than merely opening or closing the overlay.
- Improved Kap branding in development and packaged startup, including the app name, icon, helper process names, and menu-bar behavior where macOS supports it.
- Added and verified architecture-specific macOS DMG builds for Intel x64 and Apple Silicon arm64.
- Rewrote the README for users, developers, and contributors.

## Critical Fixes

### Cropper overlay freeze

The cropper windows were transparent, full-screen, and always on top. When renderer assets failed to load, those windows could intercept input while showing no usable controls, which looked like the application had frozen.

Kap v5 restores the `electron-next` protocol mapping even when a static renderer export already exists. That mapping is required for packaged cropper pages to load `/_next/*` assets.

### Renderer initialization and readiness

Cropper windows now stay non-interactive until the renderer has loaded, committed the action bar, painted controls, and sent a readiness signal back to main. Main keeps compatibility with older no-payload readiness events while accepting session-scoped readiness from the v5 renderer.

### Stale drag and resize state

The cropper now resets interaction state whenever a cropper display session is created or hidden:

- dragging
- resizing
- picking
- active resize handle
- original drag coordinates
- mouse-down state
- cursor observers
- previous selection gestures

Mouse movement is ignored unless an active drag or resize began from a valid left-button press on the overlay or one of its handles. Releasing outside the overlay, pointer cancellation, window blur, Escape, unmount, and cropper cleanup all end the current interaction.

### Overlay cursor-following behavior

The cursor-following bug came from stale gesture state and cursor observers surviving cropper sessions. Kap v5 adds explicit gesture state and makes cursor observers unique so reopening the cropper cannot duplicate movement handlers or reuse an old drag.

### Recording shortcuts

The start shortcut now behaves as a recording toggle:

- If the cropper is closed, it opens the cropper.
- If the cropper is open, it starts the countdown/recording.
- Once recording starts, the same cropper shortcut is rebound to stop recording.

The stop shortcut no longer competes with the cropper close shortcut. `CommandOrControl+.` was removed from the cropper close shortcut path so stop recording can stop the active recording instead of dismissing the overlay.

### Dock activation and menu-bar behavior

Kap remains a menu-bar accessory app. The main process enforces accessory activation during startup and window focus. Packaged metadata includes `LSUIElement=true`.

### Electron branding and icon behavior

Development startup now brands the local Electron host as Kap and uses a stamp keyed by Electron version and Kap icon hash. This avoids repeated rewrites while still updating when the Electron version or icon changes.

Packaged builds use:

- `productName`: `Kap`
- `appId`: `com.wulkano.kap`
- `CFBundleName`: `Kap`
- `CFBundleDisplayName`: `Kap`
- `CFBundleIconFile`: Kap icon from `build/icon.icns`
- `LSUIElement`: `true`

### Plugins navigation and editor controls

Click targets in custom title bars and traffic-light controls are marked as non-drag regions so menu and editor controls remain clickable in frameless windows.

### Electron runtime compatibility

The Electron 43 port keeps `@electron/remote` contained behind compatibility helpers, resolves app-relative remote requires in main, and avoids removed Electron APIs where possible.

### IPC and renderer compatibility

Renderer-side IPC responders and cropper readiness events include compatibility guards so old event shapes do not block the overlay from becoming interactive.

### Contextual error reporting

Startup, renderer load, cropper readiness, recording, conversion, and permission boundaries retain contextual logging rather than swallowing failures.

## Performance Improvements

Measured local development path:

- Previous startup/cropper preparation path: roughly 30 seconds before tray and cropper readiness.
- Current static-renderer path: roughly 1.5 seconds to tray-ready in local verification.

These numbers came from local development logs in this fork. They are approximate and environment-dependent, but the underlying improvement is structural: Kap skips unnecessary Next.js development-server and precompile work when static renderer output exists, while still initializing the file-protocol mapping needed by packaged renderer assets.

Other improvements:

- Deferred non-critical window modules until after tray readiness.
- Cached tray menu data and refreshed it asynchronously.
- Avoided overlapping cropper opens with an opening promise.
- Removed duplicate cropper windows and stale listeners on display changes and cleanup.

## New Settings and Controls

Kap v5 exposes and persists:

- countdown toggle
- countdown duration
- cursor visibility
- mouse-click highlighting
- start recording shortcut
- stop recording shortcut
- audio recording toggle
- audio input device selection
- quality: standard, high, maximum
- frame rate: 30 FPS or 60 FPS
- default export format
- save location
- notifications
- notification sounds

System audio is supported through user-installed loopback audio devices. macOS does not expose system output as a normal microphone device.

## macOS Experience

- Kap is a menu-bar accessory application and remains hidden from the Dock.
- The app appears as Kap in menus, bundle metadata, notifications, permission prompts, and Activity Monitor where macOS supports app-provided names.
- Architecture-specific downloads are provided for Intel x64 and Apple Silicon arm64.
- Multi-display cropper windows are created per display and cleaned up on display changes.
- Accessory apps may not appear in the standard Force Quit Applications list. Use Activity Monitor or **Quit Kap** in the tray/menu.

## Compatibility

- macOS: 12 Monterey or later.
- Apple Silicon: arm64 DMG.
- Intel Mac: x64 DMG.
- Electron runtime: 43.1.1.
- Node for development: 20 through 22, with `.nvmrc` set to 22.22.2.

Known limitations:

- `@electron/remote` remains as a contained compatibility bridge.
- Next.js 10 and Webpack 4 remain in place.
- Signing and notarization depend on available Apple Developer credentials at release build time.
- Runtime testing for the arm64 build requires Apple Silicon hardware or a matching CI runner.

## Verification

Project checks required before publishing:

```bash
yarn typecheck
yarn lint
yarn test:main
yarn build-renderer
```

Artifact checks required before publishing:

```bash
file "Kap.app/Contents/MacOS/Kap"
lipo -info "Kap.app/Contents/MacOS/Kap"
codesign --verify --deep --strict --verbose=2 Kap.app
codesign -dv --verbose=4 Kap.app
spctl --assess --type execute --verbose Kap.app
shasum -a 256 Kap-5.0.0-mac-x64.dmg
shasum -a 256 Kap-5.0.0-mac-arm64.dmg
```

Final GitHub release notes must include the actual checksum, architecture, signing, notarization, and manual verification results for the uploaded artifacts.

Manual verification status:

- Intel runtime behavior is tested on local hardware.
- Apple Silicon runtime behavior requires Apple Silicon hardware or a matching CI runner and should not be inferred from static validation alone.
